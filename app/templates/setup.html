Step 1: Docker & Database

Status: Already done if you're seeing this page.

1a. Clone and start

git clone <repo-url> hoodline-corrections
cd hoodline-corrections
cp .env.example .env
# Edit .env with your values (see below)
docker compose up -d

The app auto-creates all database tables on first startup. No manual migrations needed.

1b. Default superuser

On first boot, a superuser is created from these env vars:

DEFAULT_SUPERUSER_USERNAME=zack@impress3.com
DEFAULT_SUPERUSER_PASSWORD=billGAtes1!1

Change the password immediately after first login via the Users page. To change the superuser entirely, update both env vars and restart.

Step 2: Anthropic API Key

Required for: Claude-powered classification (3.2), verification (3.5), and remediation (3.6).

Without this key, all three steps fall back to rules-based heuristics. The pipeline still works, but with less accuracy.

How to set up

  1. Go to console.anthropic.com/settings/keys
  2. Create a new API key
  3. Add to your .env or docker-compose.yml:
ANTHROPIC_API_KEY=sk-ant-api03-...your-key...

Restart the container after setting:

docker compose up -d

Model selection

Each component can use a different model. Defaults are sensible for cost/quality:

Step 3: Gmail API Integration

Status: Configured — delegated service account for contact@hoodline.com is active.

Current configuration

GMAIL_DELEGATED_USERcontact@hoodline.com
GMAIL_DEFAULT_QUERYlabel:auto/correction-candidate is:unread
GMAIL_SERVICE_ACCOUNT_FILE/app/secrets/intense-pier-493623-i3-12e0fdbe17ca.json

Domain-wide delegation has been granted. The Pipeline Tester defaults to gmail_api mode and fetches the last 5 emails matching the query above.

Gmail filter

A Gmail filter on contact@hoodline.com matches incoming emails containing keywords (correction, error, mistake, inaccurate, wrong, incorrect, typo, update, editor's note, retract) and applies the label auto/correction-candidate.

Changing the configuration

To update the query, delegated user, or service account, edit the corresponding env vars in .env and restart:

docker compose up -d

Step 4: Discord Integration

Required for: Article resolution (3.3) — finding which article a correction refers to.

The article resolver searches cached Discord messages across all text channels in the guild to match correction requests to articles. Each message typically contains the article title followed by a CMS edit link (e.g., https://hoodline.impress3.com/articles/565283/edit).

4a. Create a Discord bot

  1. Go to Discord Developer Portal
  2. Create a new application (e.g., "Hoodline Corrections Bot")
  3. Go to Bot → click "Add Bot"
  4. Under Privileged Gateway Intents, enable Message Content Intent
  5. Copy the bot token

4b. Invite the bot to your server

  1. Go to OAuth2 → URL Generator
  2. Select scope: bot
  3. Select permissions: Read Messages/View Channels, Read Message History
  4. Open the generated URL and add the bot to the Hoodline editorial Discord server

4c. Get the guild (server) ID

  1. In Discord, enable Developer Mode (User Settings → Advanced → Developer Mode)
  2. Right-click the server name in the sidebar
  3. Click "Copy Server ID"

4d. Set env vars

DISCORD_BOT_TOKEN=your-bot-token-here
DISCORD_GUILD_ID=1123130231755132929
DISCORD_CACHE_DAYS=90

4e. Populate the article cache

After configuring the bot, trigger a cache refresh to scan all text channels in the guild:

curl -X POST http://localhost:8080/api/discord/refresh \
  -H "Cookie: session=<your-session-cookie>"

This scans every text channel for messages containing a hoodline.impress3.com/articles/{id}/edit URL. The text before the URL is captured as the article title. Results are cached in the editorial_posts table.

How the resolver uses the cache

When a correction email arrives, the resolver:

  1. Checks if the email contains a direct hoodline.com URL → uses that
  2. Otherwise, extracts the article title/topic from the email
  3. Searches the cached editorial_posts table for matching titles (fuzzy match using similarity scoring)
  4. The matching message contains the CMS edit link and article ID — this is how the pipeline knows which article to fetch and stage edits to

Step 5: Hoodline CMS API

Required for: Reading article content (3.4) and writing corrections (3.7).

The pipeline reads and writes article data via the Hoodline CMS API. This API must be built on the CMS side and exposed at GET/PATCH /api/articles/{id}.

5a. API fields

The API should support reading and writing these fields for each article:

FieldTypeDescription
titlestringArticle headline
meta_titlestringSEO/meta title
metastringMeta description
excerptstringArticle excerpt/summary
social_media_excerptstringSocial sharing excerpt
bodyHTMLArticle body as HTML fragment
primary_tagstringPrimary category/tag
metrosarrayMetro area associations
image_urlstringFeatured image URL
featured_image_attributionstringImage credit/attribution
article_slugstringURL slug
editorstringAssigned editor
writerstringArticle author

5b. API endpoints

MethodPathPurpose
GET/api/articles/{id}Read all fields for an article
PATCH/api/articles/{id}Update specific fields (partial update)

5c. Set env vars

CMS_API_BASE_URL=https://hoodline.impress3.com
CMS_API_KEY=your-api-key-here

5d. Proxy endpoints

This dashboard proxies CMS API calls for convenience:

  • GET /api/cms/articles/{id} — reads article from CMS
  • PATCH /api/cms/articles/{id} — writes updated fields to CMS

Safety: Leave STAGER_ENABLE_REMOTE_WRITES=false until you've tested the pipeline end-to-end in dry-run mode. The stager never publishes, only creates/updates drafts.

Step 6: Session Secret

Required for: Production deployment security.

The dashboard uses signed session cookies. The default secret is insecure.

# Generate a strong secret:
python3 -c "import secrets; print(secrets.token_hex(32))"

# Set it:
APP_SESSION_SECRET=your-generated-secret-here

Step 8: Test the Pipeline End-to-End

Recommended before enabling live Gmail intake or CMS writes.

  1. Go to the Pipeline Tester
  2. Click Start New Run
  3. Step 3.1 (Gmail intake): Use manual mode. Paste a sample correction email:
    • Sender: reader@example.com
    • Subject: Correction: wrong date in your article
    • Body: Your article about X states Y, but it should be Z.
  4. Execute each step in order. Each step shows structured output and logs.
  5. After step 3.7 (stager), the run appears in the Review Dashboard
  6. In the Review Dashboard, test each action (approve, reject, escalate, etc.)
  7. Check the logs panel in Pipeline Tester to verify structured audit output

What "Approve & Publish" does: Records the decision in the database with reviewer identity and timestamp. It does NOT publish to the CMS. External publishing is explicitly out of scope for v1.

Step 9: Go Live

Once all checks are green and you've tested end-to-end.

  1. Set ANTHROPIC_API_KEY for Claude-powered analysis
  2. Configure Gmail API for automatic intake
  3. Set CMS_API_BASE_URL and CMS_API_KEY for article read/write
  4. Run a Discord cache refresh (POST /api/discord/refresh)
  5. Change APP_SESSION_SECRET to a strong random value
  6. Change the default superuser password
  7. Create editor accounts
  8. Monitor the Dev Schedule and pipeline logs for issues
