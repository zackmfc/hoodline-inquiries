Step 1: Docker & Database

Status: Already done if you're seeing this page.

1a. Clone and start

git clone <repo-url> hoodline-corrections
cd hoodline-corrections
cp .env.example .env
# Edit .env with your values (see below)
docker compose up -d

The app auto-creates all database tables on first startup. No manual migrations needed.

1b. Default superuser

On first boot, a superuser is created from these env vars:

DEFAULT_SUPERUSER_USERNAME=zack@impress3.com
DEFAULT_SUPERUSER_PASSWORD=billGAtes1!1

Change the password immediately after first login via the Users page. To change the superuser entirely, update both env vars and restart.

Step 2: Anthropic API Key

Required for: Claude-powered classification (3.2), verification (3.5), and remediation (3.6).

Without this key, all three steps fall back to rules-based heuristics. The pipeline still works, but with less accuracy.

How to set up

  1. Go to console.anthropic.com/settings/keys
  2. Create a new API key
  3. Add to your .env or docker-compose.yml:
ANTHROPIC_API_KEY=sk-ant-api03-...your-key...

Restart the container after setting:

docker compose up -d

Model selection

Each component can use a different model. Defaults are sensible for cost/quality:

Step 3: Gmail API Integration

Required for: Automatic email intake from contact@hoodline.com (step 3.1 in gmail_api mode).

Without this, you can still use manual mode in the Pipeline Tester to paste emails by hand.

How to set up

  1. Create a Google Cloud project (or use an existing one)
  2. Enable the Gmail API in the project
  3. Create a Service Account:
    • IAM & Admin → Service Accounts → Create
    • Download the JSON key file
  4. Set up domain-wide delegation:
    • In Google Workspace Admin → Security → API controls → Domain-wide delegation
    • Add the service account's client ID
    • Scope: https://www.googleapis.com/auth/gmail.readonly
  5. Create a Gmail filter on contact@hoodline.com:
    • Match keywords: correction, error, mistake, inaccurate, wrong, incorrect, typo, update, editor's note, retract
    • Apply label: auto/correction-candidate
  6. Set env vars:
GMAIL_DELEGATED_USER=contact@hoodline.com
GMAIL_SERVICE_ACCOUNT_FILE=/run/secrets/gmail-service-account.json
# OR inline:
GMAIL_SERVICE_ACCOUNT_JSON={"type":"service_account",...}
GMAIL_DEFAULT_QUERY=label:auto/correction-candidate is:unread

Mount the JSON key file into the container via Docker volume or secrets.

Step 4: Hoodline CMS Access

Required for: Article fetching from CMS (3.4) and draft staging (3.7).

4a. Fetcher session cookie (optional)

Allows the fetcher to probe CMS edit URLs. Without it, the fetcher only reads the public article page.

  1. Log into {{ cms_base_url }} as a bot/service account
  2. Copy the session cookie from browser dev tools
  3. Set: CMS_FETCHER_SESSION_COOKIE=wordpress_logged_in_xxx=value

This cookie will expire periodically. An automated refresh mechanism is planned but not yet built.

4b. Draft stager credentials

Enables the stager to write drafts to the CMS via WordPress REST API.

  1. In WordPress, create a dedicated Application Password:
    • Users → your bot account → Application Passwords
    • Create new, copy the generated password
  2. Set env vars:
CMS_STAGER_USERNAME=hoodline-bot
CMS_STAGER_APP_PASSWORD=xxxx xxxx xxxx xxxx xxxx xxxx
STAGER_ENABLE_REMOTE_WRITES=true
STAGER_MODE=wp_rest

Safety: Leave STAGER_ENABLE_REMOTE_WRITES=false until you've tested the pipeline end-to-end in dry-run mode. The stager never publishes, only creates/updates drafts.

Step 5: Session Secret

Required for: Production deployment security.

The dashboard uses signed session cookies. The default secret is insecure.

# Generate a strong secret:
python3 -c "import secrets; print(secrets.token_hex(32))"

# Set it:
APP_SESSION_SECRET=your-generated-secret-here

Step 6: Create Editor Accounts

Required for: Review dashboard access.

  1. Go to the Users page
  2. Create accounts for Eddie and any other editors who need review access
  3. Assign the admin role (gives access to Review Dashboard, Pipeline Tester, and Setup Guide)
  4. Share credentials securely
RoleAccess
superuserEverything, including managing other superusers
adminReview Dashboard, Pipeline Tester, Users (can only create/edit user role), Setup Guide
userHome, Dev Schedule only

Step 7: Test the Pipeline End-to-End

Recommended before enabling live Gmail intake or CMS writes.

  1. Go to the Pipeline Tester
  2. Click Start New Run
  3. Step 3.1 (Gmail intake): Use manual mode. Paste a sample correction email:
    • Sender: reader@example.com
    • Subject: Correction: wrong date in your article
    • Body: Your article about X states Y, but it should be Z.
  4. Execute each step in order. Each step shows structured output and logs.
  5. After step 3.7 (stager), the run appears in the Review Dashboard
  6. In the Review Dashboard, test each action (approve, reject, escalate, etc.)
  7. Check the logs panel in Pipeline Tester to verify structured audit output

What "Approve & Publish" does: Records the decision in the database with reviewer identity and timestamp. It does NOT publish to the CMS. External publishing is explicitly out of scope for v1.

Step 8: Go Live

Once all checks are green and you've tested end-to-end.

  1. Set ANTHROPIC_API_KEY for Claude-powered analysis
  2. Configure Gmail API for automatic intake
  3. Set CMS credentials and STAGER_ENABLE_REMOTE_WRITES=true for draft staging
  4. Change APP_SESSION_SECRET to a strong random value
  5. Change the default superuser password
  6. Create editor accounts
  7. Monitor the Dev Schedule and pipeline logs for issues
